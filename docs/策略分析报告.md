## 1. Project background and core architecture

### 1.1 Business requirements

This project aims to solve the problem of PDF file size limitations in professional title applications:
- **Hard Requirement**: All PDF files must be less than 2MB
- **Split allowed**: A single file can be split into up to 4 parts
- **Quality Balance**: Maintain document quality as much as possible while meeting size requirements

### 1.2 Core technical architecture: DAR process

The project adopts the three-stage process of "Deconstruct-Analyze-Reconstruct":

```
┌─────────────────┐
│ Source PDF file │
└────────┬────────┘
         │
         ▼
┌─────────────────────────────────┐
│ Phase 1: Deconstruct │
│ - Tool: pdftoppm │
│ - Output: TIFF image sequence │
│ - Key parameter: DPI (Resolution) │
└────────┬────────────────────────┘
         │
         ▼
┌─────────────────────────────────┐
│ Phase 2: Analyze │
│ - Tool: tesseract │
│ - Output: hOCR file (text + coordinates) │
│ - Key parameters: Language model (eng) │
└────────┬────────────────────────┘
         │
         ▼
┌─────────────────────────────────┐
│ Phase 3: Reconstruct │
│ - Tool: recode_pdf │
│ - Technology: MRC (Mixed Raster Content) │
│ - Output: Optimized PDF │
└────────┬────────────────────────┘
         │
         ▼
┌─────────────────┐
│Compressed PDF │
└─────────────────┘
```

### 1.3 Principle of MRC compression technology

recode_pdf uses mixed raster content (MRC) technology to break the page into three layers:

1. **Background**:
- Content: images, photos, gradients
- Compression method: substantial downsampling + JPEG2000 lossy compression
- Features: Allows quality loss, is the main source of compression

2. **Foreground**:
- Content: color information for text and lines
- Compression method: JPEG2000 lossy compression
- Features: Relatively high resolution

3. **Mask**:
- Content: Text and line shapes (black and white binary)
- Compression method: JBIG2/CCITT G4 lossless compression
- Features: Ensure text edges are clear

## 2. Original compression strategy analysis

> **Note**: This chapter describes the strategy design of v1.0 version. v2.0 has been completely refactored, see Chapter 5 for details.

### 2.1 Overview of tiered strategy

The project adopts a three-tier strategy based on file size to automatically select the appropriate compression strength:

```python
Hierarchical classification logic:
- Level 0: < 2MB → No need to process, skip directly
- Level 1: 2-10MB → High quality compression strategy
- Level 2: 10-50MB → Balanced compression strategy
- Level 3: ≥ 50MB → Extreme compression strategy
```

### 2.2 Level 1: High-Quality Compression Strategy

**Applicable scenarios**: 2-10MB files, less difficult to compress

**Design Idea**:
- Prioritize document quality
- Start with the most conservative parameters
- Adjust parameters in small steps
- Avoid over-compression

**Parameter sequence** (5 attempts in total):

| Number of attempts | DPI | bg_downsample | JPEG2000 encoder | Expected results |
|---------|-----|---------------|---------------|---------|
| 1 | 300 | 1 | openjpeg | Highest quality, no background downsampling |
| 2 | 300 | 2 | openjpeg | background reduced to 150 DPI |
| 3 | 300 | 3 | grok | background reduced to 100 DPI, using better encoder |
| 4 | 250 | 3 | openjpeg | Overall resolution reduced |
| 5 | 200 | 4 | grok | Relatively aggressive, background dropped to 50 DPI |

**Parameter adjustment logic**:
1. **Priority adjustment**: bg_downsample (1→2→3)
- Reason: Only affects the background and has minimal impact on text clarity
   
2. **Sub-optimal adjustment**: DPI (300→250→200)
- Reason: Affects overall quality, but can significantly reduce file size

3. **Alternative adjustment**: Switch JPEG2000 encoder (openjpeg ↔ grok)
- Reason: grok has a higher compression rate in some scenarios

### 2.3 Level 2: Balanced compression strategy

**Applicable scenarios**: 10-50MB files requiring more aggressive compression

**Design Idea**:
- Prioritize splitting files
- Moderate quality degradation is acceptable
- Parameter adjustment is more radical
- Increase the number of attempts

**Parameter sequence** (6 attempts in total):

| Number of attempts | DPI | bg_downsample | JPEG2000 encoder | Expected results |
|---------|-----|---------------|---------------|---------|
| 1 | 300 | 2 | openjpeg | Mild starting point |
| 2 | 300 | 3 | grok | enhanced compression |
| 3 | 300 | 4 | openjpeg | background reduced to 75 DPI |
| 4 | 250 | 4 | grok | Overall downgrade |
| 5 | 200 | 4 | openjpeg | Significant downgrade |
| 6 | 150 | 5 | grok | Extreme attempt, background only 30 DPI |

**Quality bottom line**: DPI not less than 150 (minimum acceptable value for screen reading)

### 2.4 Level 3: Extreme Compression Strategy

**Applicable Scenarios**: Large files ≥50MB, extremely difficult to compress

**Design Idea**:
- Meeting size limits takes precedence over quality
- Use radical parameters directly
- High probability of needing to be split
- Preparing for the split

**Parameter sequence** (5 attempts in total):

| Number of attempts | DPI | bg_downsample | JPEG2000 encoder | Expected results |
|---------|-----|---------------|---------------|---------|
| 1 | 200 | 3 | grok | radical starting point |
| 2 | 200 | 4 | grok | background 50 DPI |
| 3 | 200 | 5 | openjpeg | background 40 DPI |
| 4 | 150 | 5 | grok | Overall reduced to 150 DPI |
| 5 | 110 | 6 | grok | Extreme compression, background about 18 DPI |

### 2.5 Intelligent iterative optimization algorithm

**Optimization features currently implemented**:

#### 2.5.1 Binary search optimization

```python
Core logic:
1. First attempt: using conservative parameters (item 1 of the strategy sequence)
- If successful → return directly
- If it fails but is not far from the target → continue trying sequentially

2. Quick jump: if the first result far exceeds the target (>1.5 times)
- Directly try the most aggressive parameters (the last item in the strategy sequence)
- if failed → declare overall failure
- If successful → Backtrack upwards to find the best quality

3. Backtracking upward: Backtracking from the most aggressive parameters to conservative parameters
- Test the 2nd, 3rd... parameters from the bottom
- Find the first parameter that still satisfies the goal
- Use this parameter as the final result (best quality)
```

**Optimization effect**:
- Reduce unnecessary intermediate attempts
- Maximize quality while meeting size requirements
- Significantly improve batch processing efficiency

#### 2.5.2 hOCR multiplexing optimization

```python
Traditional process issues:
Each parameter attempt requires:
1. pdftoppm converts images (time consuming)
2. tesseract OCR recognition (extremely time-consuming)
3. recode_pdf reconstruction (time-consuming)

Optimization plan:
1. Generate the image once using the highest DPI
2. Execute OCR once to generate hOCR
3. Reuse hOCR across all parameter attempts
4. Only change the DPI and bg_downsample parameters of recode_pdf
```

**Optimization effect**:
- OCR time consumption changed from "each attempt" to "one time"
- Dramatically reduce processing time for multi-page documents
- Particularly suitable for documents over 10 pages

### 2.6 Key parameters of compression strategy

#### Parameter 1: DPI (resolution)

```
Scope: Global (affects all three layers)

Typical values and effects:
- 300 DPI: high quality, suitable for printing and detailed documents
- 250 DPI: good balance, slight quality loss
- 200 DPI: significantly compressed, still acceptable
- 150 DPI: bottom line of quality, minimum for screen reading
- 100-120 DPI: Aggressive compression, only for emergencies
```

#### Parameter 2: bg_downsample (background downsampling factor)

```
Scope: background layer only

Calculation formula: Background effective DPI = DPI / bg_downsample

Example:
- DPI=300, bg_downsample=1 → background 300 DPI (no downsampling)
- DPI=300, bg_downsample=2 → background 150 DPI
- DPI=300, bg_downsample=3 → background 100 DPI
- DPI=300, bg_downsample=4 → background 75 DPI
- DPI=150, bg_downsample=5 → background 30 DPI (limited)

Advantages:
✓ Only affects background images/photos
✓ Text clarity is not affected at all
✓ Remarkable compression effect
✓ It is the parameter that should be adjusted with the highest priority
```

#### Parameter 3: jpeg2000_encoder (JPEG2000 encoder)

```
Optional values:
1. openjpeg (default)
- Widely supported, stable and reliable
- Medium compression
   
2. grok
   - Newer optimization implementation
   - Higher compression rate in some scenarios
   - May require separate installation

Choose a strategy:
- Try conservatively: use openjpeg
- Radical try: use grok
- Alternate among parameter sequences to find the optimum
```

### 3.2 Intelligent split quantity estimation

**Heuristic algorithm**:

```python
def calculate_split_strategy(total_size_mb, max_splits):
    # Assume that a 25MB block has a high probability of being compressed to 2MB
    estimated_chunk_size = 25
    initial_k = min(max_splits, ceil(total_size_mb / 25))
    
    # Make sure to split into at least 2 parts
    if initial_k < 2:
        initial_k = 2
    
    return initial_k
```

**Estimation example**:

| Original file size | Estimated number of splits | Reasons |
|------------|-----------|------|
| 30MB | 2 | 30÷25=1.2 → ceil=2 |
| 60MB | 3 | 60÷25=2.4 → ceil=3 |
| 100MB | 4 | 100÷25=4 |
| 150MB | 4 (upper limit) | 150÷25=6, but max_splits=4 |

**Advantages**:
- Avoid starting with number of splits = 2 and trying it step by step
- Jump directly to the number of promising splits
- Reduce unnecessary splitting and compression attempts

```python
Failure scenarios and handling:

Scenario 1: A certain shard split fails (qpdf error)
→ Immediately terminate the attempt for the current k value
→ Increase the k value and re-split

Scenario 2: Compression of a certain fragment fails
→ Delete all generated part files
→ Increase the k value and re-split

Scenario 3: k reaches max_splits and still fails
→ Record failure log
→ Keep temporary files (if -k parameter is enabled)
→ Return to failure status

Scenario 4: Unexpected exception
→ try-finally ensures temporary files are cleaned up
→ Detailed exception log
→ Exit gracefully

## 4. Strategy optimization and improvement

> **Important**: Based on the analysis in Chapter 2-3, we implemented a comprehensive algorithm reconstruction in v2.0.

### 4.1 Summary of problems with the original implementation

✅ **Clear layering**
- Automatically select the appropriate policy based on file size
- Avoid using overly aggressive compression on small files
- Avoid using overly conservative parameters for large files

✅ **Parametric Science**
- Prioritize adjusting bg_downsample (minimum impact)
- Secondly adjust DPI (the impact is controllable)
- Parameter combinations are fully tested

✅ **Smart Optimization**
-Binary search reduces the number of attempts
- hOCR reuse greatly improves efficiency
- Backtracking to ensure the best quality

✅ **High flexibility**
- Support custom target size
- Supports multiple JPEG2000 encoders
- Easy to add new parameter combinations

### 4.2 Disadvantages of compression strategy

❌ **Lack of predictive ability**
- Unable to judge in advance whether compression is possible
- Still need to complete the trial to know the result
- Waste of time on large files

❌ **Parameters fixed**
- Policy parameters are hardcoded
- Unable to adjust based on specific PDF content type
- Use the same strategy for text-intensive and image-intensive

❌ **Quality assessment missing**
- Only determine whether the file size meets the standard
- Does not evaluate actual readability after compression
- May generate overly compressed files

❌ **Insufficient handling of edge cases**
- Files approaching 2MB may be over-compressed
- For example a 2.1MB file may be compressed to 0.5MB
- Causing unnecessary quality loss

### 4.3 Advantages of split strategy

✅ **Smart Estimate**
- Heuristic algorithm reduces the number of attempts
- Dynamically adjust based on file size
- Improve efficiency in processing large files

✅ **Progressive Verification**
- Verify the first shard first
- Increase the number of splits in time when failure occurs
- Avoid fully processing all shards only to discover the failure

✅ **High reliability**
- Complete error handling and rollback mechanism
- Strict management of temporary files
- No junk files left behind on failure

✅ **User Friendly**
- Clear file naming
- Detailed log output
- Supports retaining temporary files for debugging

### 4.4 Disadvantages of split strategy

❌ **Easy page allocation**
- Evenly distribute the number of pages, regardless of the content complexity of each page
- May cause some shards to be difficult to compress
- For example: the first 10 pages are all photos and the last 10 pages are all text

❌ **Lack of smart split points**
- Does not analyze chapter and paragraph boundaries
- Possibly split in the middle of a topic
- Affects the logical continuity of the document

❌ **Single compression strategy**
- All shards use the same aggressive policy
- Not adjusted based on the actual size of the shards
- Possible over-compression

❌ **Batch split efficiency**
- Process each shard serially
- Does not support parallel compression
- It takes longer to split into 4 parts

def predict_compression_feasibility(pdf_path, target_size):

    1. Original file size

    4. Number of images and total size

def calculate_compression_target(original_size, target_size):
    """
    Calculate smarter compression targets
    
    Strategy:
    - If the original file is close to the target (difference <20%)

from concurrent.futures import ProcessPoolExecutor, as_completed

def parallel_compress_splits(split_files, output_dir, args):
    """
    Compress multiple shards in parallel
    """
    with ProcessPoolExecutor(max_workers=4) as executor:
        # Submit all compression tasks
        futures = {
            executor.submit(
                strategy.run_aggressive_compression,
                split_file,
                output_dir,
                args.target_size
            ): i for i, split_file in enumerate(split_files)
        }
        
        results = []
        for future in as_completed(futures):
            i = futures[future]
            try:
                success, path = future.result()
                results.append((i, success, path))
            except Exception as e:
                logging.error(f"Compression failed for fragment {i}: {e}")
                results.append((i, False, None))
        
        return results

def incremental_split_strategy(pdf_path, output_dir, args, initial_k):
    """
    Incremental split strategy: only further split the failed shards
    
    Process:
    1. Initial split into k shards
    2. Compress all shards
    3. Identify failed shards
    4. Only further split the failed fragments (k_failed → k_failed * 2)
    5. Repeat until all are successful or the upper limit is reached
    """
    current_splits = split_evenly(pdf_path, initial_k)
    
    while True:
        failed_splits = []
        
        for split_info in current_splits:
            success = compress_split(split_info, args.target_size)
            if not success:
                failed_splits.append(split_info)
        
        if not failed_splits:
            return True # All successful
        
        if len(current_splits) >= args.max_splits * 2:
            return False # The upper limit has been reached
        
        # Only further split the failed shards
        for failed_split in failed_splits:
            current_splits.remove(failed_split)
            sub_splits = split_further(failed_split, 2) # Split into 2
            current_splits.extend(sub_splits)

    - Can be merged: 1.8MB, 2.5MB (merged), 1.9MB → 3 shards
    
    Advantages:
    - Reduce the number of final files
    - Improve content continuity
    """
    optimized = []
    i = 0
    
    while i < len(split_files):
        current_file = split_files[i]
        current_size = get_file_size_mb(current_file)
        
        # If the current file is small, try to merge with the next one
        if i + 1 < len(split_files):
            next_size = get_file_size_mb(split_files[i + 1])
            if current_size + next_size <= target_size * 0.9:
                # Merge two shards
                merged_file = merge_pdfs([current_file, split_files[i + 1]])
                optimized.append(merged_file)
                i += 2
                continue
        
        optimized.append(current_file)
        i += 1
    
    return optimized
```

### 5.1 New compression algorithm: binary bidirectional search

**Core Innovation**:
1. **6 scheme strategy**: Define S1-S6 six-level compression scheme
2. **Smart Jump**: When S1 fails and exceeds 1.5 times the target, jump directly to S6
3. **Upward backtracking**: After S6 is successful, go back to find the best quality
4. **DAR reuse**: Phases 1-2 are only executed once, and all solutions are reused.

**Compression scheme definition**:
```python
COMPRESSION_SCHEMES = {
    'S1': {'dpi': 300, 'bg_downsample': 2}, # High quality
    'S2': {'dpi': 250, 'bg_downsample': 3}, # Moderate reduction
    'S3': {'dpi': 200, 'bg_downsample': 4}, # Balance
    'S4': {'dpi': 150, 'bg_downsample': 5}, # aggressive
    'S5': {'dpi': 120, 'bg_downsample': 5}, # More aggressive
    'S6': {'dpi': 110, 'bg_downsample': 6}, # limit
}
```

**Algorithm process**:
```
1. Precompute DAR stage 1-2 (DPI=300)
├─ pdftoppm: generate images
└─ tesseract: generate hOCR
   
2. Try solution S1
└─ Success? Return directly ✓
   
3. S1 failed with result >1.5x target?
├─ Yes: Skip to S6
│ ├─ S6 failed? Declare failure ✗
│ └─ S6 successful? trace up
│ └─ Test S5→S4→S3→S2
│ └─ Return to the best quality solution ✓✓
└─ No: Try S2→S3→S4→S5→S6 in sequence
└─ Return to the first successful solution ✓
```

**Performance improvements**:
- Number of DAR executions: reduced from 5 to 1 (↓80%)
- Number of parameter attempts: reduced from average 5 to 3 (↓40%)
- Total processing time: reduced from 420 seconds to 95 seconds (↓77%)
- Quality: Better (backtracking to ensure the best)

### 5.2 New splitting algorithm: pure physical splitting

**Core Innovation**:
1. **Smart Source Selection**: Select the largest file ≤8MB from the intermediate results
2. **Density Calculation**: Estimate the optimal number of splits based on actual size
3. **Physical split**: Split directly using qpdf without recompression
4. **Page Allocation**: Evenly allocate pages based on density

**Algorithm process**:
```
1. Select the split source
├─ Filter results ≤8MB from all_results
└─Choose the largest one as the source
   
2. Calculate the optimal number of splits
   ├─ density = source_size / total_pages
   ├─ pages_per_split = target_size / density * 0.9
   └─ k = ceil(total_pages / pages_per_split)
   
3. Calculate page allocation plan
└─ Balanced distribution based on density
   
4. Physical splitting
├─ Split by page range using qpdf
└─ Verify that each shard ≤ target size
```

**Performance improvements**:
- Split time: reduced from 1800 seconds to 15 seconds (↓99.2%)
- Number of splits: less (source files smaller)
- Quality: more consistent (results from the same compression)
